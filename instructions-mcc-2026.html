Each student reserves exactly one assignment.

Reservation happens through a Pull Request. The solution is built in your own project repository, tested clearly, and submitted here as repository and demo links.

Name and email are mandatory. The reservation format is `Taken by Full Name <email@example.com>; Stack: Chosen stack`.
Change exactly one line. Students must not edit descriptions, scores, scripts, workflows, HTML pages, or another student's topic.
The reservation is official after automatic merge. Copying the claim line is not enough. Open a PR; GitHub Actions checks it and automatically merges the first valid reservation.

How to reserve a topic

Reservations are first in, first served. The first valid PR that passes automatic validation owns the topic and updates the public site.

Watch the localhost demo: choose an available topic, copy the claim line, and open a PR. Valid reservations are checked and merged automatically.

  1. Open `mcc-2026.html`, find a topic, and check that it is `Available`.
  2. Click `Reserve`, enter your full name and email. Both are required.
  3. Choose your stack, then click `Copy claim line`; the correct line is copied, but this is not an official reservation yet.
  4. Click `Edit on GitHub`; GitHub opens `CLOUD_TOPICS_2026.md` directly for editing.
  5. Replace only your topic line with the copied line, then click `Propose changes` and create a Pull Request.
  6. Wait for the automatic validation. The first valid PR is merged automatically; if another PR reserved the same topic first, your PR will stay unmerged and must be changed.

One student can reserve only one assignment. Validation rejects repeated names or emails.

Valid reservation PR

Change exactly one line in `CLOUD_TOPICS_2026.md`: replace only `(Available)` with `(Taken by Full Name <email@example.com>; Stack: Chosen stack)`. If your topic is already reserved, use `Check your reservation`, choose/change the stack, and open a correction PR that edits only your same topic line.

Invalid PRs

Do not edit `mcc-2026.html`, do not rename files, do not remove `Cloud focus` or `Score`, do not duplicate the title, and do not put two students on one topic.

After GitHub Actions merges your valid PR automatically, use `Check your reservation` on the MCC page with the same full name or email to reopen your topic details. Reservation PRs are processed one at a time, so first-in, first-served is decided atomically by GitHub Actions.

How to build the assignment

Do not build the solution inside this catalogue repository. Create your own project repository.

Project repository

Create a new repo, for example `mcc-2026-topic-01-full-name`, and place your source code there.

Clear README

Explain your stack, setup, `.env` variables, local run commands, and deployment steps.

Cloud/MCC services

Use meaningful cloud services: auth, database, storage, serverless functions, jobs, logs, push notifications, or deployment.

AI log

If you use AI tools, keep `AI-log.txt` with the important prompts and help you used.

How to test the solution

Before submission, the project must be reproducible and demonstrable.

Local testing

  • The project starts with the README commands.
  • No secrets are committed to source code.
  • Main forms, lists, and workflows work.

Cloud testing

  • Deployment URL or API URL works.
  • Auth, database, storage, or functions are connected.
  • Logs show request success or useful errors.

MCC/mobile testing

  • The mobile app opens on emulator or device.
  • Sync, offline cache, or push notifications are tested if relevant.
  • Photos, location, or sensors are tested with real or documented mock data.

Evidence

  • Add screenshots, a short video, or a demo URL.
  • Document known limitations.
  • Check that public repos contain no keys or tokens.

How and where to submit

Final submission happens in this repository, but only as a link to your own project repository.

  1. When the project is ready, open `SUBMISSIONS_2026.md`.
  2. Find the line with your topic ID.
  3. Change only that line from `Not submitted` to the submission format.
  4. Open a new Pull Request for final submission.
  5. The PR must include the repo link, demo URL or `N/A`, and the same name/email used for reservation.

Grading

Scores range from 0 to 30 and are edited only by the instructor.

Students only read scores

Scores appear in the catalogue as `Not graded` or `N/30`. Students do not change them.

What is checked

Functionality, cloud usage, code quality, testing, documentation, and final demonstration.

What is rejected

PRs with many changed lines, missing email, second reservations, non-working repos, or public secrets.
